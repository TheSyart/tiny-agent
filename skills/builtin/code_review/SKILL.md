# Code Review — 代码审查技能

## 触发词
用户说以下内容时加载本技能：
- 帮我 review 代码 / 审查代码 / 看看这段代码
- 代码有什么问题 / 有什么可以改进的
- 检查安全漏洞 / 性能优化 / 代码质量

## 审查维度（优先级顺序）

### 1. 正确性（Critical）
- [ ] 逻辑是否正确，边界条件是否处理
- [ ] 返回值/错误处理是否完整
- [ ] 并发场景是否线程安全
- [ ] 数据类型是否匹配，类型转换是否安全

### 2. 安全性（Critical）
- [ ] 输入验证：是否对用户输入进行清洗/验证
- [ ] SQL 注入：是否使用参数化查询
- [ ] XSS：HTML 输出是否转义
- [ ] 命令注入：shell 调用是否使用参数列表（非字符串拼接）
- [ ] 敏感信息：密码/Token/Key 是否硬编码在代码中
- [ ] 路径遍历：文件路径是否经过规范化验证

### 3. 性能（High）
- [ ] 循环内是否有重复 I/O 操作（N+1 查询）
- [ ] 大数据集是否使用生成器/流式处理
- [ ] 是否有不必要的同步阻塞（asyncio 场景）
- [ ] 缓存策略是否合理

### 4. 可读性（Medium）
- [ ] 变量/函数命名是否清晰表达意图
- [ ] 复杂逻辑是否有注释说明 *为什么*
- [ ] 函数长度是否合理（超过 50 行考虑拆分）
- [ ] 重复代码是否提取为公共函数

### 5. 健壮性（Medium）
- [ ] 异常是否被合理捕获和处理（避免 bare `except:`）
- [ ] 资源（文件、连接）是否用 context manager 管理
- [ ] 超时/重试机制是否有限制（避免无限重试）

## 工作流程

1. **快速扫描**：通读代码，理解整体结构和意图
2. **安全检查**：优先排查安全问题（注入、敏感信息）
3. **逻辑验证**：逐行核对关键逻辑，尤其是条件分支和错误处理
4. **性能评估**：识别潜在瓶颈
5. **风格建议**：最后提可读性改进（非强制）

## 输出格式

```markdown
## 代码审查结果

### 🔴 严重问题（必须修复）
- [行号] 问题描述 → 修复建议

### 🟡 改进建议（推荐修复）
- [行号] 问题描述 → 修复建议

### 🟢 优点
- 值得保留/学习的良好实践

### 总结
一句话概括整体代码质量和主要关注点。
```

## 示例

用户：帮我 review 这段 Python 函数
```python
def get_user(id):
    sql = f"SELECT * FROM users WHERE id = {id}"  # ← SQL 注入风险
    result = db.execute(sql)
    return result[0]  # ← 未处理空结果
```

审查输出：
- 🔴 SQL 注入：使用 f-string 拼接 SQL，改为 `db.execute("SELECT * FROM users WHERE id = ?", (id,))`
- 🔴 未处理空结果：`result[0]` 在无记录时抛 IndexError，改为 `return result[0] if result else None`
